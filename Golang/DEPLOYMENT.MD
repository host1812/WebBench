# Azure VM Deployment

This service now uses `golang-migrate` to apply and track database migrations in a managed PostgreSQL database.

The managed database is outside Docker Compose. The VM runs:

- an OpenTelemetry Collector container
- a one-shot `migrate` container
- the API container
- an nginx gateway container that terminates HTTPS

Migrations are tracked in the managed database in the `schema_migrations` table.

The schema does not require PostgreSQL extensions. UUID values are generated by the Go service before insert, so Azure Database for PostgreSQL does not need `pgcrypto` or `uuid-ossp`.

## Files On The VM

The VM only needs:

```text
/opt/books-service/
  .env
  compose.yaml
  nginx.vm.conf
  otel-collector-config.yaml
  certs/
    server.crt
    server.key
  certbot-webroot/
  migrations/
    001_init.up.sql
    001_init.down.sql
    002_seed_catalog.up.sql
    002_seed_catalog.down.sql
```

The deploy script copies these files for you. It also copies your local `.env` file to `/opt/books-service/.env`. It generates `certs/server.crt` and `certs/server.key` on the VM if they do not already exist, then attempts to replace them with a trusted Let's Encrypt certificate.

## Configure `.env`

Create a local `.env` from the committed template:

```powershell
Copy-Item .env.example .env
```

Set these production values before deploying:

```text
BOOKSVC_IMAGE=your-registry.azurecr.io/books-service-golang:latest
BOOKSVC_DATABASE_CONNECTION_STRING=postgres://user:password@your-managed-postgres-host:5432/books?sslmode=require
APPLICATIONINSIGHTS_CONNECTION_STRING=InstrumentationKey=...;IngestionEndpoint=...;LiveEndpoint=...
```

If the PostgreSQL password contains special URL characters, URL-encode it in `BOOKSVC_DATABASE_CONNECTION_STRING`.

Do not commit `.env`.

## Configure Managed PostgreSQL

Set this value in `.env` to your managed PostgreSQL connection string:

```text
BOOKSVC_DATABASE_CONNECTION_STRING=postgres://postgres:postgres@your-managed-postgres-host:5432/books?sslmode=require
```

For Azure Database for PostgreSQL, `sslmode=require` is usually required.

Also confirm the API image is correct in `.env`:

```text
BOOKSVC_IMAGE=your-registry.azurecr.io/books-service-golang:latest
```

## Configure Application Insights

Set this value in `.env`:

```text
APPLICATIONINSIGHTS_CONNECTION_STRING=InstrumentationKey=...;IngestionEndpoint=...;LiveEndpoint=...
```

The API sends OpenTelemetry traces to the local Collector over OTLP HTTP:

```yaml
BOOKSVC_TELEMETRY_OTLP_ENDPOINT: "otel-collector:4318"
```

The Collector exports those traces to Azure Application Insights using [otel-collector-config.yaml](C:/Projects/WebBench/Golang/otel-collector-config.yaml).

## Deploy

From your local machine:

```powershell
.\scripts\deploy.ps1 -VmIp <vm-public-ip>
```

The script:

- creates `/opt/books-service`
- copies `.env` as `/opt/books-service/.env`
- copies `compose.vm.yaml` as `/opt/books-service/compose.yaml`
- copies `nginx.vm.conf`
- copies `otel-collector-config.yaml`
- copies the full `migrations` directory
- creates a self-signed HTTPS certificate if one does not already exist
- runs `docker compose pull`
- runs `docker compose run --rm migrate`
- starts the HTTPS gateway with `docker compose up -d gateway`
- attempts to install Certbot and request a Let's Encrypt IP-address certificate
- keeps the self-signed certificate if Let's Encrypt cannot be configured

If migrations fail, the script stops before starting the API.

The script logs into Azure on the VM by using the VM managed identity:

```bash
az login --identity
az acr login --name <your-acr-name>
```

The managed identity assigned to the VM must have `AcrPull` permission on the registry.

The ACR name is hardcoded in `scripts/deploy.ps1` as `$AcrName`.

## HTTPS

The VM deployment uses nginx as a TLS gateway:

```text
internet -> nginx:443 -> api:8080
```

The API container is only exposed inside the Docker network. Public traffic should go through HTTPS on port `443`.

The deploy script first creates a self-signed certificate for the VM IP address:

```text
/opt/books-service/certs/server.crt
/opt/books-service/certs/server.key
```

After the gateway is running, it attempts to install Certbot and request a Let's Encrypt IP-address certificate with the required `shortlived` profile. If that succeeds, the Let's Encrypt certificate replaces the self-signed files and nginx is reloaded.

The script also creates this renewal hook:

```text
/etc/letsencrypt/renewal-hooks/deploy/books-service-nginx.sh
```

The hook copies renewed certificates into `/opt/books-service/certs` and reloads nginx.

If the Let's Encrypt step fails, the self-signed certificate remains active. Browsers and normal `curl` calls will warn until the certificate is trusted by the client.

For quick verification:

```bash
curl -k https://localhost/health
curl -k https://<vm-public-ip>/health
```

Open inbound TCP `443` in the Azure VM network security group. Open inbound TCP `80` if you want the Let's Encrypt step to succeed, because IP-address certificates are validated through the HTTP challenge path served by nginx.

If you later add a DNS name, use a normal domain certificate instead of an IP-address certificate. Domain certificates have broader client and automation support.

## Migration Commands

SSH into the VM:

```bash
ssh azureuser@<vm-public-ip>
cd /opt/books-service
```

Check the applied migration version:

```bash
docker compose run --rm migrate version
```

Apply all pending migrations:

```bash
docker compose run --rm migrate up
```

Rollback one migration:

```bash
docker compose run --rm migrate down 1
```

Rollback all migrations:

```bash
docker compose run --rm migrate down
```

Do not run rollback commands in a real environment unless deleting schema or seed data is intended.

## How Tracking Works

`golang-migrate` creates and maintains a table named:

```text
schema_migrations
```

It records:

- the latest applied migration version
- whether the database is in a dirty state

The current migration files are:

```text
001_init.up.sql
001_init.down.sql
002_seed_catalog.up.sql
002_seed_catalog.down.sql
```

When `docker compose run --rm migrate up` runs, only migrations not already recorded in `schema_migrations` are applied.

## Dirty State

If a migration fails halfway, `golang-migrate` may mark the database as dirty. Check with:

```bash
docker compose run --rm migrate version
```

Fix the database manually, then force the version only if you understand the current schema state:

```bash
docker compose run --rm migrate force <version>
```

Then rerun:

```bash
docker compose run --rm migrate up
```

## Health Check

After deployment:

```bash
curl -k https://localhost/health
```

From your local machine:

```bash
curl -k https://<vm-public-ip>/health
```

Make sure the VM network security group allows inbound TCP traffic on port `443`.

## Telemetry Verification

Check that the Collector is running:

```bash
docker compose ps otel-collector
```

The VM Compose file uses a stable versioned contrib Collector image:

```text
otel/opentelemetry-collector-contrib:0.140.0
```

Check Collector logs:

```bash
docker compose logs -f otel-collector
```

Generate a request:

```bash
curl -k https://localhost/health
```

Then open Application Insights in the Azure portal. Trace data can take a few minutes to appear.
